**Детализированный план на хакатон**

Целевой результат: за 5 дней собрать воспроизводимый MVP под требования хакатона, где пользователь вводит задачу на русском или английском, система локально через Ollama получает YAML-метаданные, валидирует их, при необходимости делает до 3 итераций уточнения, затем генерирует Lua по шаблону, прогоняет проверку и отдаёт результат. Основание для этого плана: task_hackathon.md, roadmap_hackathon.md и текущая расширенная архитектура в comprehensive_v1.mmd.

1. Зафиксировать границы MVP.
Критичное решение: не строить enterprise-архитектуру из полной диаграммы. В MVP остаются только локальная модель, FastAPI, YAML-контракт, цикл валидации, 3 шаблона Lua, sandbox, минимальный UI, docker-compose и документация. Убираются cache, RAG, DAG orchestration, audit, metrics, oracle storage, version storage и любые необязательные подсистемы. Это блокирующий шаг: без него команда расползётся по лишнему объёму.

2. Утвердить целевой pipeline.
Рабочая цепочка должна быть ровно такой: пользовательский запрос → Ollama с qwen2.5-coder:1.5b → YAML metadata → JSON Schema validation → structured feedback → повтор генерации до 3 раз или уточняющий вопрос → template selector → Lua generation → luac → Docker sandbox → ответ пользователю. Heavy-модель допустима только как ручной запасной режим, а не как обязательная часть основного пути.

3. Определить API и внутренние контракты.
Нужно зафиксировать 3 публичных сценария:
1. health-check для проверки живости сервиса
2. генерация YAML и итогового Lua по запросу
3. диалоговое уточнение, если модели не хватает входных данных

Внутренне нужно определить:
1. структуру YAML с минимальными полями operation, parameters, conditions
2. возможные поля для уточнения, если данных не хватает
3. единый формат ошибки валидации: поле, ожидаемое значение, фактическое значение, подсказка
4. единый формат результата sandbox: статус, stdout, stderr, код завершения

4. Сделать backend-каркас.
Backend должен включать:
1. HTTP-слой на FastAPI
2. клиент для Ollama
3. сервис валидации YAML по схеме
4. session state для attempts и context
5. сервис выбора шаблона
6. сервис генерации Lua
7. сервис запуска проверок

Definition of done для этого этапа: сервис стартует локально, health endpoint отвечает, ошибки возвращаются в предсказуемом формате.

5. Сделать контракт YAML и валидацию.
Это один из самых рискованных участков, потому что модель может генерировать нестабильную структуру. Поэтому здесь нужно:
1. максимально жёстко задать output contract в prompt
2. определить узкий набор допустимых operations
3. валидировать все обязательные поля
4. возвращать модели не общий текст ошибки, а структурированное сообщение
5. ограничить число повторов тремя попытками

Definition of done: на типовых примерах модель либо выдаёт валидный YAML, либо даёт корректный вопрос на уточнение, либо завершает цикл понятной ошибкой.

6. Реализовать цикл уточнения и ReAct-логику.
Логика должна быть простой:
1. первая генерация пытается сразу собрать YAML
2. если схема не проходит, backend формирует structured feedback
3. модель делает повторную попытку
4. если проблема в недостатке входных данных, пользователь получает вопрос
5. после ответа пользователя контекст сессии обновляется и цикл продолжается
6. после трёх неудач сервис завершает сценарий контролируемой ошибкой

Здесь важно не строить сложную агентную сеть. Для хакатона достаточно одного управляемого цикла, который реально работает.

7. Подготовить шаблонный слой Lua.
Нужно ограничиться тремя шаблонами из roadmap:
1. file_ops
2. http_request
3. json_parse

Template selector должен быть строгим:
1. operation маппится только на разрешённый шаблон
2. неизвестный operation даёт явную ошибку
3. прямую свободную генерацию Lua в обход шаблонов не добавлять

Definition of done: для каждого из трёх классов задач есть хотя бы один рабочий сценарий генерации.

8. Добавить проверки кода и sandbox.
Проверка идёт в два этапа:
1. синтаксическая проверка через luac
2. запуск в Docker sandbox

Требования к sandbox:
1. timeout 5 секунд
2. память 128 MB
3. без сети
4. возврат stdout и stderr
5. предсказуемый статус ошибки

Это важно не только для качества решения, но и для демонстрации безопасного локального исполнения.

9. Собрать минимальный UI.
UI не должен быть красивым любой ценой, он должен быть понятным на демо. Достаточно одного экрана с:
1. полем ввода задачи
2. отображением YAML или ошибки валидации
3. блоком уточняющего вопроса
4. выводом итогового Lua-кода
5. кнопкой запуска проверки
6. выводом результата sandbox

Приоритет у скорости сборки. Если выбор стоит между сложным frontend и работающим demo path, выигрывает demo path.

10. Упаковать запуск в один воспроизводимый сценарий.
Нужно собрать запуск так, чтобы жюри могло поднять решение одной командой. Здесь обязательны:
1. docker-compose
2. явная инструкция по установке модели
3. точный тег модели
4. параметры запуска Ollama: num_ctx 4096, num_predict 256, batch 1, parallel 1
5. отсутствие внешних AI API в runtime

Это ключевой критерий оценки по локальности и воспроизводимости.

11. Переработать существующий README.
Текущий README.md слишком краткий и пока не закрывает судейский сценарий. Его нужно расширить так, чтобы в нём были:
1. что делает проект
2. какие ограничения хакатона он закрывает
3. как скачать модель
4. как поднять всё одной командой
5. как проверить, что сервис работает
6. какие есть ограничения MVP
7. как воспроизвести демонстрационный сценарий
8. как измерить VRAM и подтвердить, что пик не превышает лимит

12. Привести архитектурную диаграмму в соответствие реальности.
Сейчас comprehensive_v1.mmd описывает более широкую систему, чем нужно для MVP. Здесь два допустимых пути:
1. упростить текущую диаграмму до реально реализуемого пайплайна
2. оставить её как расширенное видение и добавить отдельную MVP/C4-диаграмму

Рекомендация: второй вариант. Он чище, потому что не смешивает хакатонный scope и post-hackathon ambitions.

13. Подготовить тестовый контур.
Минимум нужно закрыть:
1. 5 позитивных типовых пользовательских сценариев
2. невалидный YAML
3. неизвестный operation
4. исчерпание 3 попыток
5. падение sandbox
6. проверку каждого из 3 шаблонов хотя бы на одном кейсе

Это даст не только уверенность в демо, но и материал для презентации.

14. Подготовить финальные артефакты.
К сдаче должны быть готовы:
1. открытый репозиторий
2. расширенный README
3. docker-compose и однокомандный запуск
4. C4 или MVP-диаграмма
5. демо-видео
6. презентация
7. зафиксированный сценарий показа эталонного запроса
8. подтверждение параметров модели и VRAM

**Параллельность и зависимости**

Можно делать параллельно:
1. backend-каркас и YAML-контракт
2. UI и backend после появления базового API
3. документацию и упаковку запуска после стабилизации базового pipeline
4. шаблоны и schema work, если список operations уже утверждён

Нельзя делать вразнобой:
1. template selector нельзя стабильно завершить до фиксации operation contract
2. sandbox не имеет смысла доводить до конца, пока нет хотя бы одного рабочего шаблона
3. README нельзя финализировать до определения реального способа запуска
4. диаграмму нельзя утверждать до фиксации MVP-границ

**Разбивка по дням**

1. Сегодня вечером.
Зафиксировать scope, выбрать конечный pipeline, утвердить модель, описать YAML-контракт, поднять каркас backend и подготовить минимальный docker-compose skeleton.

2. День 1.
Довести генерацию YAML, schema validation и endpoint генерации до рабочего состояния. Получить первые 3–5 примеров, где структура устойчива.

3. День 2.
Добавить ReAct-цикл, session state, endpoint уточнения и пользовательский сценарий с хотя бы одной итерацией диалога.

4. День 3.
Сделать 3 шаблона Lua, selector, luac и sandbox. На выходе должен появиться полный путь от текста до исполняемого Lua.

5. День 4.
Собрать всё вместе, сделать UI, провести end-to-end прогоны, привести README и диаграммы в порядок, подготовить видео и презентацию.

6. День 5 утром.
Прогнать эталонный сценарий, зафиксировать VRAM, проверить запуск на чистом окружении и сдавать.

**Главные риски**

1. Модель не держит YAML-структуру стабильно.
Первое решение: жёстче output contract, already-known operations и post-processing. Не расширять архитектуру раньше времени.

2. Слишком широкий scope.
Первое решение: не трогать enterprise-компоненты из полной диаграммы, пока не закрыт основной pipeline.

3. Sandbox или Lua runtime съедают время.
Первое решение: сначала добиться успешного luac и простого контейнерного запуска на одном сценарии, потом расширять.

4. README и запуск остаются “на потом”.
Первое решение: как только появляется рабочий путь, сразу фиксировать его в README.md, иначе к сдаче документация разойдётся с реализацией.

**Критерий успеха**

План считается выполненным, если жюри может локально поднять решение, ввести запрос на русском или английском, получить валидный Lua через управляемый цикл генерации и валидации, а команда может показать, что всё работает без внешних AI-сервисов и в пределах VRAM-ограничений.
